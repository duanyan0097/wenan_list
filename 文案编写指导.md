# 文案编写指导

这个文件是后续写录屏口述稿的主入口。以后先看这个文件，再根据需要去看 `cases/` 里的单个案例。

## 一句话定位

写出来的内容要像一个人正在电脑前操作，顺手把问题、判断和测试过程讲出来。

不要像正式讲课，不要像包装稿，不要像提前背好的完整稿。

## 核心目标

1. 让内容听起来真实，不像摆拍。
2. 让问题从屏幕操作里自然出现。
3. 讲清楚看起来简单的设置，为什么实际容易不稳定。
4. 讲清楚判断顺序，而不是只说最后结果。
5. 结尾必须有测试动作，证明状态是否正常。

## 推荐语气

可以有这种口语：

```text
其实这个设置很简单。
但是呢，它能用是一回事，长期稳定地用又是另一回事。
你看这个地方，很多人会以为是这里没设置好，其实不一定。
```

这种语气可以有停顿、重复、转折，但不能乱。听起来要像真实操作时说出来的，不是整理成书面稿。

## 禁止口吻

不要这样写：

```text
今天介绍一个稳定访问文件的办法。
这个办法适合很多地方使用。
我们已经把这个问题处理好了。
```

这种太完整，太像提前写好的稿子。

也不要这样写：

```text
现在给你演示一下完整流程。
第一步，第二步，第三步。
最后我们可以看到效果非常好。
```

这种像教学视频，不像真实调试。

## 正确结构

每条口述稿按这个顺序写：

```text
先说屏幕上正在看的东西
再说这个东西看起来简单在哪里
然后说实际经常出问题在哪里
接着说很多人容易误判哪里
再说应该按什么顺序检查
最后做打开、保存、复制、刷新、换电脑等测试
```

## 常用开头方式

不要用固定开场。直接从正在看的画面进入。

可以这样开：

```text
这个共享设置你看一下，其实就这几个地方。
这个文件夹本地能打开，但是换一台电脑就不一定。
这个地方很多人会先去改权限，其实顺序可能反了。
这个状态不能只看它连上了，还要看后面会不会掉。
```

## 常用转折句

```text
但是呢，这个地方容易出问题。
很多人会以为是这里，其实不一定。
这个时候不要先急着改配置。
先看它到底是网络没通，还是单台电脑自己的状态不对。
如果顺序反了，就会一直在一个地方绕。
```

## 常用验证句

```text
现在目录能打开，这只说明第一步正常。
还要打开文件保存一下，看另一边能不能刷新出来。
如果文件稍微大一点，还要复制一次，看中间会不会断。
换一台电脑再试一次，才能判断是不是单台电脑的问题。
不能只看打开一次，后面连续使用状态也要看。
```

## 单条文案模板

```text
标题：

屏幕上正在看的内容：

看起来简单的地方：

实际容易出问题的地方：

常见误判：

判断顺序：

要展示的操作：

最后测试动作：

口述稿：
```

## 示例：Windows 共享不稳定

```text
这个共享设置你看一下，其实就这几个地方。

把共享打开，权限设置一下，基本上就能用了。

但是呢，它能用是一回事，长期稳定地用又是另一回事。

很多时候不是共享本身没点对，而是每台电脑自己的状态不一样。

比如系统版本不一样，网络类型不一样，有的还保存过旧账号。它就会出现这种情况：今天这台能进，明天那台又不行。

所以这个地方不要只盯着共享设置。先看访问的那台电脑能不能到这个地址，再看网络类型、防火墙和凭据。

最后还要实际打开文件，保存一下，再换一台电脑试一次。只打开一次不能说明它后面就稳定。
```

## 示例：异地访问先判断网络

```text
这个文件夹在本地能打开，不代表换个地方也能打开。

很多人一看到异地打不开，就一直去改共享权限，或者反复输账号密码。

其实这个顺序可能反了。

两个地方网络本来没通的话，共享设置得再对也访问不到。

所以先不要急着改文件夹，先看能不能到那个地址，再看对应访问有没有通。

网络通了以后，再回头测共享目录，打开文件，保存文件，刷新另一边看状态。

这样判断才比较准，不然就会一直在权限里面绕。
```

## 写完后的检查

写完后按这几条检查：

1. 有没有像真实录屏时顺口讲出来。
2. 有没有先从画面和现象开始。
3. 有没有讲清楚常见误判。
4. 有没有讲判断顺序。
5. 有没有写最后测试动作。
6. 有没有过度完整、过度正式。
7. 有没有出现身份标签或具体领域标签。

不符合就重写，不要硬修句子。
